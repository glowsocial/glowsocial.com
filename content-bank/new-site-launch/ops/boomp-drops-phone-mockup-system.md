# Boomp Drops Phone Mockup System

This file defines the **default image system** for Boomp Drops canonical pages.

## Core decision
Every Boomp Drop page should show **one example post idea inside a phone mockup**.

The image is not decorative.

Its job is to answer:
> "What could this idea look like as an actual post?"

That keeps the pages from feeling text-only while reinforcing the website-first → email/social adaptation model.

## Default placement on the page
Place the phone mockup **inside the page body**, not as a giant hero.

### Preferred position
- after the H1
- after the short direct-answer intro
- before the deeper supporting sections

### Practical placement rule
The mockup should appear in the **upper-middle / top-third** of the article.

In page flow, that usually means:
1. H1
2. 2–5 sentences of direct setup
3. phone mockup block
4. remainder of article

## Why this placement wins
- lets the page establish the idea before showing the example
- breaks up early text before the page feels essay-heavy
- makes the mockup feel like proof-of-format, not decoration
- keeps the page primarily an answer page, not a gallery page

## One image per page by default
Default to **one phone mockup per page**.

Do not make the standard page module:
- a carousel
- multiple phone frames
- a collage
- a giant social grid

The goal is consistency and speed.

## What goes inside the phone
Each phone mockup should show **one believable social post idea** derived from the page's core angle.

### The post does not need to be fully production-final
It only needs to feel:
- plausible
- on-brand
- concrete
- readable at page size

### Minimum post ingredients
Each mockup should usually contain:
- one short headline or hook
- one supporting sentence or short block
- optional small category label
- optional small CTA line
- simple branded visual treatment

## Recommended text formula inside the mockup
Use this pattern most of the time:

### Option A — headline + support
- **headline:** 5–12 words
- **support line(s):** 1–3 short lines
- **footer/CTA:** optional short phrase

### Option B — question + answer
- **question:** 1 line
- **short answer:** 2–4 lines
- **footer:** optional reminder / CTA

### Option C — comparison / checklist
- **title:** 4–8 words
- **2–4 bullets or columns**
- **short conclusion line**

## Text-length guardrails
To keep the phone mockups readable:
- headline: ideally under 60 characters
- support copy: ideally under 180 total characters
- bullet count: usually 2–4 max
- no dense paragraphs
- no full caption walls

If the text feels like an actual Instagram caption screenshot, it is probably too much.

## What the image should feel like
The mockup should feel like:
- an example of a publishable post
- a clean artifact from the system
- simple and real, not overdesigned

It should not feel like:
- generic stock art
- decorative blog filler
- a fake social feed recreation with too much tiny UI
- a cramped template overloaded with copy

## Visual structure
### Base layout
Use a consistent reusable structure:
- centered phone frame
- branded post card inside the phone
- generous whitespace
- high-contrast readable type
- one visual hierarchy, repeated everywhere

### Suggested module sizing
- center aligned
- medium width inside article body
- large enough that headline is readable on desktop
- not so large that it dominates the page

### Default width target
Treat the block like a featured inline module, not a hero banner.

A good target is roughly:
- phone frame visually prominent
- but still clearly subordinate to the H1

## Caption rule
Default to **no caption**.

The page copy above the image already explains what the mockup is doing.

Only add a caption if it adds actual clarity, such as:
- "Example post angle"
- "One way this idea could become a social post"

If used, keep it short and quiet.

## Page-module spec
Recommended module structure on-page:

```md
# H1

2–5 sentences of direct-answer intro.

[phone mockup block]

## Supporting section
...
```

### Spacing intent
The mockup should feel attached to the early idea, not detached from it.

So:
- enough top margin to breathe after the intro
- enough bottom margin to separate it from the next subhead
- no giant dead space

## Visual variations by page type
Keep one system, but allow a few repeated post patterns.

### 1. Insight / tip card
Best for:
- owner-burden
- website-engine
- AI-search-readiness
- content-sources

Structure:
- short headline
- 1–2 explanatory lines
- small footer label or CTA

### 2. Question / answer card
Best for:
- objections
- FAQ-heavy pages
- buyer questions

Structure:
- customer-style question
- short plain-language answer

### 3. Comparison card
Best for:
- buyers-guides
- tool-vs-system pages
- decision pages

Structure:
- short title
- 2–4 rows or columns
- short takeaway line

### 4. Proof / review card
Best for:
- proof-and-reviews
- trust-signals
- some local-visibility pages

Structure:
- review line, proof snippet, or result framing
- optional small visual insert (review, before/after, screenshot)

### 5. Reminder / timing card
Best for:
- seasonal-operations
- local-visibility

Structure:
- seasonal or timing headline
- reminder / action / why-now line

## Cluster guidance
### owner-burden
Use insight cards and question/answer cards.

### trust-signals
Use proof cards and simple trust reminder cards.

### content-sources
Use tip cards and workflow-style educational cards.

### buyers-guides
Use comparison cards.

### local-visibility
Use reminder cards, platform-role cards, and occasional screenshot-backed mockups.

### website-engine
Use explainer cards and page-first workflow cards.

### objections
Use question/answer cards.

### proof-and-reviews
Use proof cards first.

### seasonal-operations
Use reminder cards and timing cards.

### ai-search-readiness
Use explainer cards, FAQ cards, and checklist/comparison cards.

## When to upgrade beyond the default template
The default is a branded post card in a phone.

Upgrade selectively when a real supporting visual makes the post stronger.

### Good upgrade cases
- review screenshot inside the mockup
- GBP screenshot inside the mockup
- before/after image inside the mockup
- simplified website screenshot inside the mockup

### Do not upgrade just because you can
If the real asset is messy, tiny, low-trust, or unreadable, stay with the clean template.

## What not to do
Do not use:
- generic stock people shaking hands
- unrelated AI art
- abstract blobs with no information value
- huge unreadable fake social UI
- long fake captions
- multiple competing visual systems across clusters

## Production rule for all 100 pages
The system should be scalable.

That means:
- one phone frame style
- one shared typography system
- a small number of post-card variants
- repeated spacing logic
- cluster-level color or accent variation only where helpful

## Minimum viable production workflow
For each page:
1. identify the page's core angle
2. reduce it to one post idea
3. choose the appropriate card type
4. write a short on-card version
5. place it in the standard phone mockup
6. insert the module after the intro

## Sanity check before shipping a mockup
Ask:
1. Does this look like a believable post idea?
2. Can someone understand it quickly?
3. Is the text readable at page size?
4. Does it reinforce the page instead of distracting from it?
5. Would the page feel weaker without it?

If the answer to #5 is no, the image is probably filler.

## Bottom line
The default image for a Boomp Drop page should be:
- **one sample post idea**
- **shown inside a phone mockup**
- **placed early in the article body**
- **used as proof of format, not decoration**

That gives the whole library a more finished, more concrete, and more publishable feel without turning the image pass into a giant custom design project.
